02 Architecture

总体架构

当前框架刻意把 LLM 生成、AST 编译、算子执行、评估反馈和记忆沉淀分开，避免 LLM 直接执行 Python 或底层函数。

LLM Agent
  -> DSL expression
  -> parse / AST
  -> validate
  -> compile
  -> KernelAdapter
  -> factor matrix
  -> evaluator
  -> factor memory
  -> next proposal

核心原则

  • LLM 只生成 DSL 表达式，不生成可执行 Python。
  • 编译器只接受 AST，不接受任意字符串 eval。
  • 公开层默认可生成，实验层必须显式 opt-in。
  • 底层算子只能通过 DSL 包装层进入系统。
  • 所有表达式必须经过执行、评估、记录、去重。

模块职责下一步关注点
registry.py统一登记 DSL 算子、层级、公开状态、参数签名和原始 kernel 映射。增加版本号、弃用标记、算子族标签。
dsl_operators.py定义 59 个 DSL 算子，包括 35 个公开层和 24 个实验层。继续从候选底层算子中提升稳定别名。

backend_consistency.py比较 NumPy、Polars、Bottleneck 后端在基准输入上的一致性。扩展到更多窗口算子和多资产面板样本。
polars_panel.py支持命名表达式批量执行，窗口聚合优先走 Polars。从单表达式执行扩展成表达式 DAG 批处理。
raw_kernel_screening.py筛选 190 个底层算子，拆成已包装、候选、阻断和排除集合。把候选提升决策接入测试和文档生成。
cli.py导出 inventory、DSL 算子清单和底层筛选报告。增加一键刷新 GitHub Pages 数据的命令。

05 Compiler

AST 编译器

ast_compiler.py 已经支持嵌套 AST 验证和执行。它是 LLM 输出和算子执行之间的安全边界。

已完成

  • 字段节点：{"field": "close"}
  • 字面量节点：{"literal": 20}
  • 算子节点：{"op": "ts_mean", "args": [...]}
  • 拒绝 raw:: 底层算子名。
  • 实验层算子必须 include_experimental=True。

下一步

  • 增加字符串 DSL grammar，例如 zscore(ts_mean(close, 20))。
  • 把 parser 输出固定为当前 AST schema。
  • 增加错误消息映射，反馈给 Agent 修改表达式。
  • 增加表达式哈希，用于因子去重和谱系记录。

06 Execution

执行与后端

当前执行层由 KernelAdapter 统一入口，底层按算子类型分发到安全数学、Bottleneck、Polars、NumPy wrapper 或原始 Numba kernel。

安全数学

log、div、pct 已走 safe_math，定义域外和除零统一输出 NaN。

窗口聚合

ts_mean、ts_std、ts_sum、ts_var 已覆盖 NumPy、Polars、Bottleneck 一致性。

批量执行

polars_panel.py 支持命名表达式批量执行；根节点聚合走 Polars，嵌套表达式回退到 AST 编译执行。
